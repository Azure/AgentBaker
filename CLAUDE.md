## Development Commands

### Build and Testing
- `make` - Generate testdata, manifest, and validate shell scripts
- `make generate` - Regenerate embedded Go testdata and manifest.json
- `make test` - Run Go unit tests (includes aks-node-controller tests)
- `make test-style` - Run linting (validate-go, validate-shell, validate-copyright-headers)
- `make build-binary` - Build the main baker binary
- `./hack/tools/bin/golangci-lint run` - Run Go linter

### Single Test Execution
- `go test ./pkg/agent/...` - Run tests for specific package
- `go test -run TestSpecificFunction ./pkg/agent/` - Run specific test function
- `GENERATE_TEST_DATA="true" go test ./pkg/agent...` - Generate testdata snapshots

### After Code Changes
**CRITICAL**: When making changes to files in `parts/` or `pkg/` directories, you MUST run `make generate` afterward to regenerate snapshot test data and ensure consistency.

### VHD and Component Testing
- `make shellspec` - Run shell script unit tests using ShellSpec
- `make shellspec-focus` - Run ShellSpec tests with focus flag
- `make validate-shell` - Validate shell scripts with shellcheck
- `make validate-components` - Validate components.json against schema
- `make test-e2e` - Run end-to-end cluster provisioning tests

### Tools and Dependencies
- `make -C hack/tools install` - Install all development tools (golangci-lint, cue, etc.)
- `make bootstrap` - Install tools and validate Git presence
- `make tidy` - Clean up Go module dependencies

## Architecture Overview

AgentBaker provisions Kubernetes nodes for AKS through two main systems:

### 1. VHD Builder System (`vhdbuilder/`)
Builds VM images using Packer with pre-installed components:
- Supports Ubuntu, Windows Server, Azure Linux, and Mariner
- Uses shared scripts from `parts/` directory
- Components defined in `parts/common/components.json`
- Packer templates in `vhdbuilder/packer/`

### 2. Runtime Services

**AgentBaker API Server** (`cmd/main.go`)
- HTTP API (port 8080) for node bootstrap data generation
- Key endpoints: `/getnodebootstrapdata`, `/getlatestsigimageconfig`
- Generates cloud-init (Linux) or PowerShell scripts (Windows)

**AKS Node Controller** (`aks-node-controller/`)
- On-node binary for configuration processing
- Commands: `provision` (run CSE scripts), `provision-wait` (status check)
- Uses protobuf configs in `aks-node-controller/proto/`

### Shared Components (`parts/`)
- **`parts.go`** - Embeds filesystem using Go embed
- **Linux**: Cloud-init artifacts in `parts/linux/cloud-init/artifacts/`
- **Windows**: PowerShell CSE scripts in `parts/windows/`
- **`components.json`** - Version definitions for all binaries/images

## Key Code Patterns

### Template Generation
- Core logic in `pkg/agent/baker.go` and `pkg/agent/bakerapi.go`
- Go templates generate OS-specific bootstrap payloads
- Data models in `pkg/agent/datamodel/` (Go structs)
- Node controller uses protobuf schemas

### Configuration Flow
1. API request → `GetNodeBootstrapData()` handler
2. AgentBaker instance processes `NodeBootstrappingConfiguration`
3. Template engine generates platform-specific scripts
4. Output: cloud-init YAML (Linux) or PowerShell (Windows)

### Component Lifecycle
- **Build time**: VHD builder installs from `components.json`
- **Runtime**: Node controller validates same component versions
- Both systems share `parts/` templates and scripts

## Development Guidelines

### Code Changes
- Follow conventional commits format for commit messages
- Prefer single-commit pull requests (use `git rebase -i` to squash)
- All changes to `parts/` or `pkg/` require running `make generate`

### Shell Scripts
- Use ShellSpec for testing shell scripts
- Run shellcheck validation via `make validate-shell`
- Scripts must work on both Azure Linux/Mariner and Ubuntu
- Source functions properly and use local variables when possible

### Go Development
- Use vanilla Go test framework
- Follow standard Go conventions
- Keep data models in `pkg/agent/datamodel/`

### Windows PowerShell
- Scripts in `staging/cse/windows/` (different from Linux `parts/`)
- Windows CSE can be downloaded as ZIP during provisioning
- Follow PowerShell best practices
- PowerShell unit tests: individual `.tests.ps1` files alongside implementation

## Testing Strategy

### Unit Tests
- Go tests: `go test ./...`
- Shell tests: `make shellspec` (Docker-based ShellSpec)
- PowerShell tests: `.tests.ps1` files in `staging/cse/windows/`
- Template validation against schemas

### Snapshot Testing
- Testdata stored in `pkg/agent/testdata/` directories
- Generated by running tests with `GENERATE_TEST_DATA="true"`
- Manually verify snapshot content represents expected output
- Search for `dynamic-config-dir` in `baker_test.go` for validation examples

### E2E Tests
- Full cluster provisioning scenarios in `e2e/`
- VHD content validation tests
- Component version consistency checks

### Required Before PRs
1. `make test-style` (linting)
2. `make generate` (if touching `parts/` or `pkg/`)
3. `make test` (unit tests)
4. Consider `make test-e2e` for significant changes
